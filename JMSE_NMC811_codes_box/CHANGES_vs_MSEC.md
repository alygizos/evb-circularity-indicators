# JMSE_NMC811_codes_box — changes vs MSEC_NMC811_codes_box

**Date:** 2026-06-21
**Scope:** Documents every change in `JMSE_NMC811_codes_box` relative to
`MSEC_NMC811_codes_box`, with a results comparison, and records the stale-value
cleanup done in the MSEC folder.

---

## 1. Summary

Three substantive updates were made in the **JMSE** box (it is now the box of record):

1. **ECPI revamped** to the raw-input / `eq2` whole-system method (math moved out of
   the CSV into the indicator). ECPI **0.137 → 0.122**.
2. **`Y_retired_years` corrected 10 → 5** (second-life duration), affecting the energy
   indicator E. E **0.6368 → 0.6383**.
3. **Sensitivity-lite engine added** (`nmc811_sensitivity_lite.py`) that recomputes
   S1/S2/S3 from the five indicator scripts. The previous MSEC tables were stale and,
   for S3-ECPI and S1-CI, physically wrong.

A supporting fix: **`ci_indicator.py` made self-contained** (it previously imported a
missing `indicators.py` and could not run standalone here).

After cleanup, the **only real numeric differences** between the two boxes are **ECPI**
(method) and **E** (`Y_retired`). Everything else is identical within Monte-Carlo noise.

### ECPI revamp detail (`eq2`)
- `circular = Σ_recovered rec_ghg_abs` (recycling-credit GHG; no virgin remainder term)
- `linear   = Σ_recovered mass_prod × virgin_ghg`
- `β = 1 − circular/linear`; `α = α_in × α_out`, with collection in α_out only.
- Inputs validated: `Σ mass_prod × virgin_ghg = 2824.18` reproduces the **old**
  `LCA_emissions_linear` exactly → masses + virgin-GHG factors confirmed.
- NMC811 result: **β 0.715, α 0.170, ECPI 0.122**. (NMC811 has no Fe/P and no Li
  recipe-excess, so `eq2` equals the whole-system value numerically.)

## 4. Stale-value cleanup performed in the MSEC folder

MSEC's stored outputs were a stale cache from an older code version. Verified by running
MSEC's *own current* code (deterministic): `total_R = 0.6198`, `CEI = 0.5683` — i.e. the
stored `R = 0.367` and `CEI = 0.540` were not reproducible.

Actions taken in `MSEC_NMC811_codes_box`:
- **`Code/cumulative_results_indicators.csv` refreshed** by re-running MSEC's own
  `cei_indicator.py` then `mre_indicator.py`. Stale rows corrected:
  - `R` 0.366601 → **0.620236**
  - `CEI` 0.540479 → **0.569249**
  - (`M`, `E` re-stamped; `PCI`, `CI`, `ECPI` already reproduced — left as-is.)
  - Pre-cleanup file saved as `cumulative_results_indicators.STALE_BAK`.
- **Stale sensitivity outputs archived** (old model: stale R, non-monotonic S3-ECPI,
  inverted S1-CI). Moved to `_STALE_archive/` in both
  `Code/sensitivity/` and `Data Collection - Code/`. `plot_sensitivity_scenarios.py`
  was kept (code, not data). MSEC has no sensitivity *generator*, so these cannot be
  regenerated in-place — the correct, current tables live in the JMSE box.

> MSEC's `ECPI` (0.137) and `E` (0.6368) are **not** stale — they are what MSEC's current
> (pre-revamp, `Y_retired=10`) code produces. They were left unchanged.

## 6. Note on `cumulative_results_indicators.csv`

This file is a **cache of outputs**, not a source of truth. A stale row means the cache
was never refreshed after a code/input change — the code is fine. De-staling = re-run the
indicator (MRE must run **last** in any batch, because PCI/CI/CEI/ECPI each rewrite the
file keeping only `[PCI,CI,CEI,ECPI]`; only MRE writes `M,R,E`).
